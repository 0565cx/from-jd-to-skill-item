# JD 转专业能力知识点 Skill

一个用于将岗位 JD 转换为**面试专业能力知识点**的 Cursor Skill，适用于中文招聘语境下的岗位分析、知识点预生成与筛选流程。

## 适用场景

当你需要根据岗位 JD 生成以下内容时，可以使用这个 Skill：

- 专业技能角度的岗位 JD解析
- 简版 JD
- 知识点清单
- 知识点筛选结果
- 校招 / 社招默认考察技能推荐

它支持两类输入方式：

1. **已有完整 JD**：直接精读岗位 JD，归纳职责与专业技能；
2. **没有 JD**：根据行业、岗位、招聘场景、经验年限，联网检索主流招聘平台与招聘官网，综合生成 JD。

## 核心能力

- 从 JD 中抽取**岗位职责**与对应的**硬技能**
- 生成符合招聘语境的**综合岗位 JD** 和 **简版 JD**
- 生成**搜索链接合集**，便于追溯 JD 来源
- 按统一列结构生成**知识点预生成 Excel**：`行业｜岗位｜技能分类｜技能｜知识点`
- 支持后续的**知识点筛选**流程
- 支持在不筛选时，直接推荐 2～5 个适合校招或社招的默认考察技能

## 生成流程

### 第一步：职责与专业技能归纳

先根据输入场景生成综合岗位 JD。

#### 场景 1：已有完整 JD

- 精读 JD
- 提炼核心职责
- 识别每项职责所需的专业技能
- 输出综合岗位 JD 与简版 JD

#### 场景 2：没有 JD

- 围绕「行业 + 岗位 + 招聘场景 + 经验年限」检索招聘平台和企业官网
- 汇总多个来源，形成一份综合岗位 JD
- 同时生成链接合集

对于制造业及其子行业，如果岗位存在明显方向分化，综合岗位 JD 会采用：

- **岗位通用职责**
- **方向职责**

并在每个职责下补充对应的专业技能。

## 制造业岗位的特殊识别规则

当你输入的行业是制造业时，这个 Skill 会有一套相对特殊的岗位JD检索和解析规则。

### 关键原则

- 岗位名称变体不能过度泛化
- 方向归纳必须从招聘资料中抽取，而不是凭岗位名主观预设
- 方向要结合：
  - 岗位名称变体
  - 职责动作
  - 工作对象
  - 主要交付物

### 岗位名称变体边界

例如“自动化技术工程师”“自动化工程师”这类岗位，变体名称原则上应保留“自动化”这一核心限定词，如：

- 自动化机械工程师
- 自动化电气工程师
- 非标自动化工程师
- 自动化设备工程师
- 自动化结构工程师
- 自动化项目工程师

不应直接泛化成更大的岗位族，例如：

- 机械工程师
- 电气工程师
- 设备工程师
- 工艺工程师

这些只能作为相邻岗位参考，不作为直接并入的变体。

### 职责交付物抽取

建议按以下结构理解岗位职责：

`JD 原文职责｜动作｜对象｜交付物｜可能方向`

例如：

- “负责机械结构设计、出图及制造过程跟进”
- “负责自动化项目原理、整体方案、详细结构设计”
- “跟进样机装配调试、解决异常问题”

这类表述会进一步抽取成结构方案、3D 模型、图纸、调试记录、问题闭环等交付物。

## 第二步：知识点预生成

Excel 固定列顺序为：


| 行业  | 岗位  | 技能分类 | 技能  | 知识点 |
| --- | --- | ---- | --- | --- |


### 生成原则

- 技能分类必须是**专业能力域名称**
- 技能要具体、可考察
- 知识点要足够细，避免过泛
- 不写软技能、上下游协作、纯工具操作类内容
- 每个技能下的知识点禁止为空

## 第三步：知识点筛选

第三步**不会自动执行**，必须由使用者明确确认后再进行。

筛选时会：

- 给技能分类、技能、知识点做核心度排序
- 标记末位知识点
- 剔除不属于本岗核心职责的内容
- 输出筛选结果表

## 不筛选时的默认推荐

如果用户明确表示暂不筛选，可以直接基于已有知识点推荐 4 个默认考察技能：

- 校招：更偏基础、必备、入门主干能力
- 社招：更偏问题诊断、方案优化、项目闭环、风险控制

输出格式只给：


| 招聘对象 | 技能分类 | 技能  |
| ---- | ---- | --- |


## 使用提示

- 适合岗位 JD 分析、专业能力知识点预生成、面试考察项整理
- 制造业岗位尤其适合用“通用职责 + 方向职责”的方式归纳
- 若岗位存在明显方向分化，先判断职责、对象和交付物，再决定知识点结构

## 示例工作流

1. 输入岗位信息或完整 JD
2. 生成综合岗位 JD、简版 JD、链接合集
3. 生成知识点预生成 Excel
4. 用户确认后再执行第三步筛选，或直接获取默认考察技能
